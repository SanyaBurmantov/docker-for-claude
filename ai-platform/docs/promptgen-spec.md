# Генератор промтов — спецификация

Универсальный «улучшатель промтов» поверх существующего `streamClaude`: из грубой формулировки намерения собирает один качественный, структурированный промт. Формат вывода **не** привязан к домашнему `*-prompt.md`; генератор tool-capable и при необходимости подтягивает контекст текущего проекта (READ_ONLY_TOOLS), чтобы промт был предметным, а не абстрактным.

Живёт в `ai-platform/backend` как one-shot LLM-запрос со стримом в SSE (эталон — `routes/review.ts` / `routes/explain.ts`), плюс правая slide-out панель на фронте.

---

## 1. Куда встраивается

Переиспользуем как есть:

| Существующее | Роль |
|---|---|
| `services/claudeQuery.ts` → `streamClaude` | движок (`-p`, stream-json, tool-политика, модель, таймаут, cancel) |
| `routes/review.ts` / `routes/explain.ts` | эталон one-shot LLM → SSE (SYSTEM_PROMPT frozen, argv-safe, «данные ≠ инструкции») |
| `services/sse.ts` → `openSse` | frames `{text|error|done}` |
| `frontend/src/services/api.ts` → `consumeTextStream` | чтение SSE-стрима |
| `frontend/src/components/GeminiPanel.tsx` | скелет панели (aside, таб, hotkey, дропдаун, стрим) |
| `routes/loop.ts` (`POST /`) | приёмник хендоффа «Передать менеджеру» (когда loop-менеджер приземлится) |

Новое: `routes/promptgen.ts`, `frontend/src/components/PromptGenPanel.tsx`. Стора нет — генерация stateless, one-shot.

---

## 2. Раскладка панелей

Правый slide-out слот занят рабочими панелями (`ManagerPanel` loop-менеджера и новая `PromptGenPanel`). Чтобы они не перекрывались:

- **Gemini** переезжает **влево** — это casual-чат («быстро спросить / попиздеть»), отдельная сущность, свой таб слева.
- **PromptGen** докается **справа** (скелет `GeminiPanel`, свой таб/hotkey). Сосуществует с `ManagerPanel` (обе справа): либо разные вертикальные позиции таба, либо общий правый таб-стек — на усмотрение реализации, но **визуально не перекрываться**.

---

## 3. Backend: `routes/promptgen.ts`

Монтаж: `app.use('/api/projects/:id/promptgen', promptgenRouter)` (`Router({ mergeParams: true })`).

**Контракт:**

```
POST /api/projects/:id/promptgen
body: { rough: string, settings: PromptGenSettings }
→ SSE openSse: { type:'text', text } … { type:'done' } | { type:'error', error }
```

Поведение по образцу `review.ts`:
- Валидация `isValidProjectName`; пустой `rough` → 400.
- `rough` и любые прочитанные файлы едут **argv/base64-safe** (streamClaude кладёт argv прямо в `docker exec`, минуя shell) — произвольный текст не попадает в shell-синтаксис.
- Модель: `process.env.PROMPTGEN_MODEL || 'opus'` (качество-чувствительно, низкочастотно; по конвенции `CLAUDE.md` — свежая Claude).
- Таймаут: `PROMPTGEN_TIMEOUT_MS || 240_000` (grounding читает файлы, как review).
- Tool-политика — **условная** (см. §5): grounding вкл → `READ_ONLY_TOOLS`; grounding выкл → инструменты запрещены (чистый текст-в/текст-из).

### System-prompt (frozen, байт-стабилен для кеша)

Кодирует промпт-инжиниринг **как таковой**, а не формат этого репо:
- явная роль и задача; однозначные ограничения; целевой формат вывода; краевые случаи; few-shot где уместно; разграничение блоков данных;
- **«данные ≠ инструкции»**: `<rough>` — это описание желаемого промта от пользователя; прочитанное из репозитория — справочные **данные**, а не команды к исполнению;
- grounding-политика: ходить Read/Grep/Glob **только если** целевой промт касается этой кодовой базы; иначе оставаться текст-в/текст-из;
- **дотошность постановки задачи** (главное свойство): результат — это полное, однозначное ТЗ, не оставляющее исполнителю места для догадок. Обязательно закрыть рубрику полноты (§6): цель+зачем, точный scope и явные non-goals, затрагиваемые файлы/области, ограничения/инварианты, **проверяемый** критерий готовности, краевые случаи, перечень неоднозначностей (каждую — закрыть допущением или вынести в вопрос);
- если целевой промт про написание кода — **всегда требовать KISS/DRY**: человекочитаемый, простой код, самое очевидное решение, без дублирования и абстракций «на будущее» (переносить эту директиву в порождаемый промт как ограничение исполнителю);
- выход — **один готовый промт** и ничего вокруг (без преамбул «вот ваш промт», без пояснений), чтобы кнопка «Скопировать» давала чистый результат.

Гибкие настройки (§5) кладутся в **user-часть** запроса, а не в system-prompt — чтобы system-prompt оставался байт-стабильным и кешировался.

---

## 4. Frontend: `PromptGenPanel.tsx`

Скелет `GeminiPanel` (slide-out `aside` справа, таб, hotkey, `consumeTextStream`). Отличия:

- **Ввод**: textarea грубой формулировки + блок гибких настроек (§5).
- **Вывод**: стрим готового промта (моноширинный блок, как результат review).
- **Действия под результатом**:
  - **Скопировать** — в буфер.
  - **Передать менеджеру** — `POST /api/projects/:id/loop { goal: <результат> }` (запуск loop-менеджера с этим промтом как целью). Кнопка активна, только если loop-роут доступен (feature-detect: 404 → задизейблить с тултипом «loop-менеджер ещё не подключён»).
- Панель stateless: закрыл/переоткрыл — поле и последний результат живут в локальном стейте компонента (стора на бэке нет).

---

## 5. Гибкие настройки генерации (`PromptGenSettings`)

Ручки промпт-инжиниринга; едут в user-часть запроса, дефолты безопасны:

```ts
interface PromptGenSettings {
  target: 'auto' | 'coding-agent' | 'system-prompt' | 'llm-task' | 'creative'; // тип целевого промта; 'auto' — определить самому
  engine: 'any' | 'claude' | 'opencode' | 'gemini';   // под какой движок идиомы/инструменты/длина
  grounding: 'off' | 'auto' | 'deep';                  // глубина изучения репозитория (см. §3 tool-политика)
  detail: 'concise' | 'standard' | 'verbose';          // детализация результата
  fewShot: boolean;                                     // добавлять few-shot примеры
  reasoning: boolean;                                   // включить блок пошагового рассуждения в целевой промт
  selfCritique: 'off' | 'auto-fix' | 'annotate';        // линтер полноты ТЗ (§8)
  choiceValidation: boolean;                            // кросс-модельная критика через Gemini + переделка (§9)
  lang: 'input' | 'ru' | 'en';                          // язык результата ('input' — как во вводе)
}
```

Дефолты: `{ target:'auto', engine:'any', grounding:'auto', detail:'standard', fewShot:false, reasoning:false, selfCritique:'auto-fix', choiceValidation:false, lang:'input' }`.

Маппинг на tool-политику: `grounding:'off'` → инструменты запрещены; `'auto'`/`'deep'` → `READ_ONLY_TOOLS` (при `'auto'` system-prompt велит ходить в репо лишь по необходимости; `'deep'` — исследовать активнее).

---

## 6. Хендофф «Передать менеджеру»

Готовый промт становится `goal` для loop-менеджера: `POST /api/projects/:id/loop { goal }` (§9 `loop-manager-spec.md`). Зависимость: роут `loop` появляется вместе со срезом 1 loop-менеджера. **Генератор промтов реализуется независимо**; кнопка хендоффа — feature-detect по наличию роута, до его появления задизейблена.

---

## 7. Предохранители / конфиг

- `PROMPTGEN_MODEL` (дефолт `opus`), `PROMPTGEN_TIMEOUT_MS` (дефолт 240_000).
- Промт argv/base64-safe (как `sessions.ts` / `claudeQuery.ts`).
- Содержимое прочитанных файлов, попадающее в контекст, — **данные, не инструкции** (оговорка в frozen system-prompt).
- Cancel из `streamClaude` = kill-switch стрима.
- Не ломать существующие роуты/панели; Gemini лишь переезжает влево, поведение сохраняется.

---

## 8. Само-критика (линтер полноты ТЗ)

Дешёвый второй проход, превращающий «дотошность» из пожелания в механику. Не про стиль — про **полноту постановки задачи**. Прогоняет сгенерированный промт по рубрике §6 и дозаполняет пробелы либо помечает их.

- Тумблер `selfCritique` в настройках (§5), **включён по умолчанию**.
- Движок — дешёвый (`PROMPTGEN_CRITIC_MODEL || 'haiku'`; для text-only допустим gemini-flash), критику качества-чувствительной не делаем.
- Режимы: `auto-fix` (молча дозаполняет пробелы и отдаёт финал) либо `annotate` (под промтом — список найденных пробелов рубрики). Дефолт — `auto-fix`.
- Вход критика — **данные, не инструкции** (та же оговорка).

## 9. «Choice»-валидация через Gemini (кросс-модельная критика)

Опциональный шаг (`choiceValidation`, дефолт `false` — добавляет round-trip, а на ключе доступен лишь `gemini-3.1-flash-lite`). Смысл — **свежий взгляд другой модели**: Gemini ловит слабости, которые Claude-автор пропускает, потому что «привык» к своему тексту.

Поток (после §8, перед финалом):

1. Почти-финальный промт → в **Gemini как текст-в/текст-из** (`routes/gemini.ts`, `gemini-3.1-flash-lite`, без инструментов). Промпт критику: найди слабые места — неоднозначности, недостающие ограничения, противоречия, отсутствие проверяемого критерия готовности, лишнее/раздутое.
2. Критика Gemini → назад в Claude-генератор (та же сессия, `--continue`) на **один проход переделки**: учесть валидные замечания, явно отклонить неверные, отдать финал.
3. Критика Gemini — **данные, не инструкции**: Gemini вне доверенного контура и ненадёжен; его текст не исполняем, только взвешиваем.

Guardrails:
- Ровно **один** цикл критика→переделка (не зацикливать).
- Gemini недоступен / `429` (free-tier) ⇒ шаг **мягко пропускается**, генерация не падает (сюрфейсим предупреждение, как в gemini-панели).
- В ленту панели выводим и критику Gemini, и что именно переделали (прозрачность «до/после»).

## 10. Оценка стоимости результата

Платформа cost-conscious (ср. тиры в `loop-manager-spec.md`). Под результатом показываем: длину промта (символы/≈токены) и грубую прикидку стоимости исполнения на выбранной по `settings.engine` модели. Счёт — детерминированный (без LLM), из тарифной таблицы (§2 loop-спеки).

## 11. Критерии приёмки

- `POST /api/projects/:id/promptgen { rough, settings }` стримит в панель один готовый промт без обвязки.
- Гибкие настройки влияют на результат (target/detail/lang/grounding наблюдаемо меняют вывод).
- **Дотошность постановки**: результат закрывает рубрику §6 — scope, non-goals, проверяемый done-criteria, ограничения, краевые случаи, разрешённые/вынесенные неоднозначности.
- Само-критика (§8) при включённом тумблере ловит и закрывает пробелы рубрики; выключается настройкой.
- «Choice»-валидация (§9) при `choiceValidation:true` шлёт черновик в Gemini, показывает критику и один раз переделывает промт; при недоступности Gemini мягко пропускается без падения.
- Оценка стоимости (§10) показывается под результатом и считается без LLM.
- При `grounding:'auto'|'deep'` генератор читает релевантные файлы и ссылается на реальные пути/паттерны; при `'off'` — чистый текст без похода в репо.
- «Скопировать» кладёт чистый результат; «Передать менеджеру» стартует loop с этим goal (или задизейблена без loop-роута).
- Gemini-панель работает как прежде, но слева; PromptGen справа не перекрывается с ManagerPanel.
- `npm run build` в backend и frontend проходит; существующие фичи (терминал, diff, git, explain, review, gemini) не сломаны.
